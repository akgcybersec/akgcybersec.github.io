I've been meaning to rewrite [RdpThief](https://github.com/0x09AL/RdpThief) for a while.
The original is a great teaching tool. It shows how to lift RDP credentials by hooking
`CryptProtectMemory` inside `mstsc.exe`, but the code hasn't aged well and the first
version I shipped still wrote captures to disk. This rewrite is the one I actually want
on an engagement: [Better-RDPThief](https://github.com/akgcybersec/Better-RDPThief).

> Before anything else: this is a proof-of-concept for authorized red team engagements
> and defensive research. Read the README's disclaimer. If you're using it, you already
> know the rules.

Here's what changed and why.

## The problem: capture path and exfil both matter

The original RdpThief hooks `CryptProtectMemory` and reads the password buffer just
before mstsc re-protects it in memory. That works for one code path, but credentials
also show up through saved-credential reads and username parsing. Different mstsc
versions and auth flows (NLA vs. legacy, saved creds vs. typed) hit different subsets.

Better-RDPThief hooks three APIs through a single **Microsoft Detours** transaction on
`DLL_PROCESS_ATTACH`:

- `CredReadW`: target, username, password from saved credentials
- `CredUIParseUserNameW`: parsed `domain\user`
- `CryptProtectMemory`: password buffer before protection

Each wrapper calls the original transparently. mstsc doesn't know we're there.

The bigger shift is exfil. The old build appended to `%TEMP%\msrdc.cache`. That's easy
to test and easy to find. This version writes nothing to disk from inside mstsc.
Credentials go out over a localhost named pipe using direct NT syscalls.

```
mstsc.exe ──[tsclient.dll]──► \\.\pipe\BetterRDPThief ──► implant (in memory) ──► C2
              pipe CLIENT          localhost only           pipe SERVER
```

The DLL is always the client. Your in-memory implant is always the server.

## Staging: username and password arrive separately

This is the annoying bit. Target, username, and password often land on different hooked
calls at different times. I kept the two-stage model:

1. **Stage**: as each hook fires, drop the captured value into globals (`gTarget`,
   `gStagedUser`, `gStagedPass`).
2. **Flush**: when all three are present, build one UTF-16 frame and send it over the
   pipe. Dedup runs first so mstsc auth retries don't spam identical captures.

The dedup is a last-seen exact match. Cheap, good enough.

## Hell's Gate pipe I/O: three syscalls, no Win32 file APIs

From inside mstsc I only need to open a pipe, write one frame, and close the handle.
So the syscall resolver pulls SSNs for exactly three stubs at runtime:

- `NtCreateFile`
- `NtWriteFile`
- `NtClose`

SSNs are parsed from clean `ntdll` stubs. API names stay rolling-ADD encoded in
`obf_enc.h` and get decoded on the stack before resolution, then wiped with `obfZero()`.

Wire format is simple: `[ULONG byteLen][UTF-16 payload]`. Connect, write, disconnect.
If the implant isn't listening yet, the client retries briefly on `STATUS_PIPE_BUSY`.
No persistent handle left open in mstsc.

I'm not pretending this defeats a competent analyst. They'll dump the process and
reconstruct the table. It drops Win32 `CreateFile`/`WriteFile` telemetry from the
capture path and keeps network I/O out of the hook DLL entirely.

## No plaintext API names in the binary

Same scheme as before: rolling-ADD encoding with an 8-byte key. Every DLL and API name
("CryptProtectMemory", "advapi32.dll", "CredReadW", etc.) lives as an encoded blob.
At runtime, `obfDecodeA()` / `obfDecodeW()` decode into a stack buffer, `LoadLibrary` +
`GetProcAddress` resolve the API, then `obfZero()` wipes the buffer.

Two consequences:

- No IAT entries for the hooked APIs. Nothing to grep for in the import table.
- Nothing sits decoded in memory longer than resolution takes.

## Lab testing without an implant

`ReadSecrets_Pipe/` is a separate build in the same repo: a named-pipe listener you
run locally while you inject `tsclient.dll` into mstsc. Same pipe name as production
(`\\.\pipe\BetterRDPThief` by default, changeable in `config.h`). Useful for proving
the hook path works before you wire the server into your agent.

## What it doesn't do

Being honest about the limits:

- **No AV evasion beyond string obfuscation and syscall pipe I/O.** Injection into
  mstsc still gets behavioral attention. YARA and Sigma rules exist for credential-hook
  patterns. EDR bypass is a separate problem stack.
- **No injection helper included.** Ship your own: sRDI, `CreateRemoteThread`, whatever
  your delivery chain expects. Inject into **mstsc.exe**, not a standalone loader process.
- **x64 only.** No 32-bit mstsc in 2026, don't care.
- **mstsc only.** The modern RDS client (`msrdc.exe`) uses different code paths. Not
  covered yet.

## OPSEC summary

| Concern | Approach |
|---|---|
| Disk artifacts in mstsc | None. Pipe client only. |
| API telemetry in mstsc | Syscall pipe I/O; hooked APIs resolved via encoded strings. |
| Network from mstsc | None. Localhost pipe only. |
| Duplicate noise | Staging + dedup before send. |

Restart a MongoDB pod without persistent storage and the data is gone. **Volumes** solve that by attaching storage that outlives any single pod instance.

Kubernetes storage has three moving parts:

1. **PersistentVolume (PV)**: the actual storage asset (admin-provisioned).
2. **PersistentVolumeClaim (PVC)**: the app's request for storage (developer-created).
3. **StorageClass (SC)**: dynamic PV provisioning template.

## Storage requirements

Real persistence needs:

1. Storage independent of pod lifecycle.
2. Storage reachable from any node (for most workloads).
3. Storage that survives cluster crashes.

**PersistentVolumes are cluster-scoped**, not namespaced. Any namespace can bind a PVC to an available PV (subject to access modes).

## Local vs remote storage

| Type | Survives node loss? | Survives cluster crash? | Use for DBs? |
|------|---------------------|-------------------------|--------------|
| Local (hostPath, local volume) | No | No | No |
| Remote (EBS, GCE PD, NFS, Ceph) | Yes | Yes | Yes |

Local volumes tie data to one node. Node dies, data is unreachable. For databases, use remote block or file storage. PVs must exist **before** pods that consume them (unless StorageClass provisions dynamically).

## Admin vs developer roles

- **Cluster admin**: provisions storage backends, creates PVs or StorageClasses.
- **Developer**: writes Deployments/StatefulSets with PVC references. Does not need to know whether the disk is EBS or NFS.

## PersistentVolumeClaim flow

```
Pod spec references PVC
        ↓
PVC requests size + accessMode (e.g. ReadWriteOnce)
        ↓
Kubernetes binds PVC to matching PV (or StorageClass provisions one)
        ↓
Volume mounted into pod container
```

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-wide">
  <img src="/resources/kubernetes/a7525b97bc074fefa9425fd6ddc75d21.png" alt="PVC binding to PV and mounting into pod" style="display:block; margin:0 auto; max-width:100%; height:auto;" />
</figure>

PVCs are **namespace-scoped**. The PV they bind to is cluster-scoped.

### Example PVC + pod mount

```yaml
# pvc.yaml
apiVersion: v1
kind: PersistentVolumeClaim
metadata:
  name: mongodb-pvc
spec:
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 1Gi
---
# snippet in deployment pod template
      volumes:
      - name: mongo-data
        persistentVolumeClaim:
          claimName: mongodb-pvc
      containers:
      - name: mongodb
        volumeMounts:
        - name: mongo-data
          mountPath: /data/db
```

```bash
kubectl apply -f pvc.yaml
kubectl get pvc
kubectl get pv
kubectl describe pvc mongodb-pvc
```

## StorageClass: dynamic provisioning

A **StorageClass** tells Kubernetes how to create PVs on demand when a PVC appears.

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-wide">
  <img src="/resources/kubernetes/377c86f311cd43278fa6dc58a282d097.png" alt="StorageClass YAML example" style="display:block; margin:0 auto; max-width:100%; height:auto;" />
</figure>

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-wide">
  <img src="/resources/kubernetes/61911f7fbe2948b087f78234abb2dace.png" alt="StorageClass dynamically provisioning PV from PVC claim" style="display:block; margin:0 auto; max-width:100%; height:auto;" />
</figure>

```yaml
apiVersion: storage.k8s.io/v1
kind: StorageClass
metadata:
  name: standard
provisioner: kubernetes.io/gce-pd   # cloud-specific
parameters:
  type: pd-standard
```

PVC references the class:

```yaml
spec:
  storageClassName: standard
  accessModes:
    - ReadWriteOnce
  resources:
    requests:
      storage: 5Gi
```

```bash
kubectl get storageclass
kubectl describe storageclass standard
```

## StatefulSets and headless services

**Deployments** are for stateless apps. **StatefulSets** give pods:

- Stable hostname (`pod-0`, `pod-1`, ...)
- Stable persistent volume per replica
- Ordered rollout and scaling

Use StatefulSets for Elasticsearch, MongoDB replicasets, MySQL with replication. Or run the database outside Kubernetes entirely (still a valid choice).

For direct pod-to-pod communication in a StatefulSet, use a **headless service** (`clusterIP: None`). DNS returns individual pod IPs instead of a single virtual IP. Needed when replicas are not interchangeable (primary vs replica).

```yaml
apiVersion: v1
kind: Service
metadata:
  name: mysql-headless
spec:
  clusterIP: None
  selector:
    app: mysql
  ports:
    - port: 3306
```

> Lab notes from authorised assessments and training environments. Do not point these techniques at systems you do not own or have written permission to test.

When Nmap shows Kubernetes ports open to the internet, you are not looking at "maybe containers." You are looking at a management plane someone forgot to firewall.

## Port map cheat sheet

```
Port(s)          Service              Risk if exposed
─────────────────────────────────────────────────────────────
6443             API Server           Cluster control (with creds or misconfig)
10250            Kubelet (auth)       Pod exec, logs (auth-dependent)
10255            Kubelet (read-only)  Anonymous pod enumeration (legacy)
10259            kube-scheduler       Info leak, debug endpoints
2379             etcd                 Full cluster state / secrets
30000-32767      NodePort             Direct service access
```

Keep this table handy during external recon. One open port changes the entire engagement scope.

## External enumeration workflow

### 1. Port scan

```bash
nmap <target> -sT -p- -T4
nmap <target> -sC -p6443,10250,10255,10259 -T4
```

Typical finds: **10250** (Kubelet API), **10259** (kube-scheduler), **6443** (API server).

### 2. Probe the API server

Even without credentials, the API server often leaks information:

```bash
curl -ks https://<master-ip>:6443
curl -ks https://<master-ip>:6443/metrics
curl -ks https://<master-ip>:6443/.well-known/openid-configuration | jq
```

Look for cluster IP (often `10.96.0.1`), issuer URLs, and supported auth methods.

`/metrics` is more useful for long-term monitoring than smash-and-grab red teams, but it reveals usage patterns and component health.

### 3. List pods (requires credentials or misconfiguration)

With valid credentials or anonymous access misconfigured away in older clusters:

```bash
curl -ks https://<master-ip>:6443/api/v1/pods | jq
```

This dumps pod names, namespaces, container images, environment variables, mounted secrets, and internal IPs. Gold for targeting.

## Anonymous kubelet read path (port 10255)

On older or misconfigured worker nodes, the **read-only kubelet API** on port **10255** exposes pod metadata without authentication:

```
Attacker ──HTTP──► Worker :10255
                      ├── /runningpods/
                      ├── /pods/
                      ├── /logs/
                      └── /configz
```

```bash
curl -sk http://<worker-ip>:10255/pods/
```

## What to do with findings

| Finding | Next step |
|---------|-----------|
| 6443 + anonymous/misconfigured RBAC | API abuse (create pods, read secrets) |
| 10255 open | Enumerate pods, hunt for creds in env vars |
| 10250 open | Test authenticated kubelet API |
| NodePort in 30000+ range | Hit the service directly |
| Dashboard on NodePort | Token theft or default creds |

Kubernetes is an open-source container orchestration framework. It manages containerised applications across environments where a single host and a single `docker run` stop being enough.

Two trends pushed us here: monoliths breaking into microservices, and containers becoming the default packaging format. When you have dozens of services spread across hundreds of containers, you need something that handles scheduling, networking, recovery, and scaling without you babysitting every process.

What Kubernetes gives you in practice:

- **High availability**: pods die; the cluster replaces them.
- **Scalability**: replicate workloads across nodes.
- **Disaster recovery**: declarative config you can re-apply after a rebuild.

This post covers the mental model. The next one gets you a cluster on your laptop.

## Nodes and pods

A **node** is a server (physical or virtual) in the cluster. Worker nodes run your application workloads.

A **pod** is the smallest deployable unit in Kubernetes. It wraps one or more containers. In most cases you run one application container per pod.

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-sm">
  <img src="/resources/kubernetes/a58e3ea99bcf4824988187542f619b92.png" alt="Two pods on a single worker node" style="display:block; margin:0 auto; max-width:100%; height:auto;" />
</figure>

Kubernetes assigns each pod its own IP on a virtual cluster network. The pod gets the IP, not the individual container inside it. Pods talk to each other by pod IP.

Pods are **ephemeral**. They die when the node runs out of resources, when the container crashes, or when you roll out a new version. A replacement pod comes up with a **new IP**. That is fine for stateless apps. It is painful for databases if clients cache the old address.

</figure>

## Services: stable addresses for unstable pods

A **Service** is a stable IP and DNS name in front of a set of pods. The service outlives any single pod instance. When pods recycle, the service endpoint stays the same.

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-sm">
  <img src="/resources/kubernetes/04889d8ae9b4476e85175679e347ed6d.png" alt="Service providing a permanent IP in front of ephemeral pods" style="display:block; margin:0 auto; max-width:100%; height:auto;" />
</figure>

- **External service**: exposes the workload outside the cluster (web apps, APIs).
- **Internal service**: cluster-only access (databases, internal APIs).

## Ingress: domain names instead of IP:port

For a browser-facing app you usually want `https://app.example.com`, not `http://10.0.0.5:30080`. **Ingress** maps hostnames and paths to services. Traffic hits the ingress controller, which forwards to the right internal service.

## ConfigMap and Secret

Applications need configuration: database URLs, feature flags, API endpoints. Without Kubernetes you bake these into images or rebuild when they change.

A **ConfigMap** holds non-sensitive config as key-value data. Mount it into a pod as environment variables or files. Change the ConfigMap, restart the pod, done.

```
DB_URL = mongodb-service
```

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-md">

Do not put credentials in a ConfigMap. Use a **Secret**. Same idea, but values are stored base64-encoded (not encryption; just not plain text in etcd by default).

## Volumes

Restart a database pod without persistent storage and you lose the data. **Volumes** attach storage to pods so data survives pod death. Backends can be local disk or cloud block storage.

## Deployments vs StatefulSets

When a pod dies, something has to recreate it. A **Deployment** is the blueprint: how many replicas, which image, which labels. The deployment owns ReplicaSets; ReplicaSets own pods.

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-md">
  <img src="/resources/kubernetes/b5c72fcce4324f57aeba6ba34cf4d68c.png" alt="Deployment replicating pods across nodes with a load-balancing service" style="display:block; margin:0 auto; max-width:100%; height:auto;" />
</figure>

Deployments work for **stateless** apps. Databases have identity and data ordering problems. Replicating MongoDB or MySQL with a plain Deployment can corrupt data. **StatefulSets** give pods stable network identities and ordered rollout. Use StatefulSets (or run the database outside the cluster) for stateful workloads.

## Control plane and worker nodes

Clusters split into **control plane** (master) nodes and **worker** nodes.

```
┌─────────────────────────────────────────────────────────┐
│                 CONTROL PLANE (master)                   │
│  API Server → Scheduler → Controller Manager → etcd      │
└──────────────────────────┬──────────────────────────────┘
                           │ kubectl / API
┌──────────────────────────▼──────────────────────────────┐
│                    WORKER NODE                           │
│  Kubelet → Container Runtime → Pods → Service            │
│  KubeProxy (pod-to-pod routing)                          │
└─────────────────────────────────────────────────────────┘
```

### Worker node processes

Every worker runs three components:

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-md">
  <img src="/resources/kubernetes/13e8e19430b649c0b15bffa60c9b675d.png" alt="Worker node: container runtime, kubelet, kube-proxy" style="display:block; margin:0 auto; max-width:100%; height:auto;" />
</figure>

- **Container runtime** (usually containerd, historically Docker): runs containers.
- **Kubelet**: talks to the API server, starts pods on this node.
- **Kube-proxy**: routes traffic to the correct pod backends.

### Control plane processes

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-wide">
  <img src="/resources/kubernetes/77e994da411c44828f67fe8a5851798e.png" alt="Master node: API server, scheduler, controller manager, etcd" style="display:block; margin:0 auto; max-width:100%; height:auto;" />
</figure>

1. **API Server**: cluster gateway. Every `kubectl` command and controller loop goes through it. Authentication and authorization happen here.
2. **Scheduler**: picks which node gets a new pod based on resources and constraints.
3. **Controller Manager**: watches cluster state. Pod crashed? Triggers a replacement.
4. **etcd**: key-value store holding cluster state. The source of truth.

A small cluster might look like two masters and three workers. Masters need less CPU/RAM; workers run the actual workloads.

<figure style="text-align:center; margin: 1.25rem 0;" class="blog-figure-md">
  <img src="/resources/kubernetes/0addfd051d894a9ea613f8e9156199ce.png" alt="Example cluster with two master and three worker nodes" style="display:block; margin:0 auto; max-width:100%; height:auto;" />
</figure>

Adding a node: provision a server, install the node components, join it to the cluster.

## Layers of abstraction

```
Deployment  →  manages ReplicaSet
ReplicaSet  →  manages Pod replicas
Pod         →  abstraction over Container(s)
```

Everything below the container image is Kubernetes' job. You declare desired state in YAML; controllers reconcile reality to match.

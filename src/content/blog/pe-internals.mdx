Most malware development courses tell you to study the PE format. Few explain *why* it matters
for offensive work beyond "you need to understand it." Here's the version I wish I had when I started:
practical, attacker-focused, with the parts that actually bite you in engagements called out clearly.

## What a PE actually is

A Portable Executable is a file the Windows loader can read, map into memory, and run. That covers
`.exe`, `.dll`, `.sys`, shellcode loaders, anything the OS knows how to execute. The format dates
to the early NT days and has barely changed at the structural level, which is both a blessing
(everything documented) and a curse (decades of quirks to navigate).

The mental model that helps most: **a PE is two different things depending on where it lives.**

- **On disk (raw):** positions are measured as file offsets from byte 0. These are called *file pointers*.
- **In memory (virtual):** positions are measured as *Relative Virtual Addresses* (RVAs), offsets from
  the image base the loader chose.

The same field in the same struct can mean completely different things depending on whether you're
reading a file off disk or a loaded image. This is the #1 source of confusion when you first write
a loader or try to parse a PE programmatically. RVA ≠ file offset. Always check which context you're in.

## The header chain

Think of PE headers as road signs. Each header tells you where the next relevant structure lives.
The chain is: DOS Header → NT Headers → Optional Header → Data Directories → Section Table.

### DOS Header

The file starts with `IMAGE_DOS_HEADER`. Almost all fields in it are legacy MS-DOS baggage you'll
never use. There's one field that matters:

```c
typedef struct _IMAGE_DOS_HEADER {
    WORD e_magic;    // "MZ" (4D 5A) - the signature that identifies a PE
    // ... lots of DOS-era fields nobody cares about ...
    LONG e_lfanew;   // file offset of the NT Headers - this is the one you want
} IMAGE_DOS_HEADER;
```

`e_lfanew` is the pointer to the real header. Everything before it is irrelevant for our
purposes, except `e_magic` (`MZ`), which is the fast check for "is this a PE at all?"

In your loader code, step one is always: validate the magic, follow `e_lfanew`.

### NT Headers

```c
typedef struct _IMAGE_NT_HEADERS64 {
    DWORD                   Signature;       // "PE\0\0" (50 45 00 00)
    IMAGE_FILE_HEADER       FileHeader;
    IMAGE_OPTIONAL_HEADER64 OptionalHeader;
} IMAGE_NT_HEADERS64;
```

`IMAGE_FILE_HEADER` contains the fields you'll actually check at runtime:

```c
typedef struct _IMAGE_FILE_HEADER {
    WORD  Machine;              // architecture: 0x8664 for x64, 0x14C for x86
    WORD  NumberOfSections;     // how many sections follow the headers
    DWORD TimeDateStamp;        // compile timestamp, useful for threat intel
    WORD  Characteristics;      // executable, DLL, etc.
    // ...
} IMAGE_FILE_HEADER;
```

The `Characteristics` field tells you if you're dealing with a DLL (`IMAGE_FILE_DLL = 0x2000`)
vs an executable (`IMAGE_FILE_EXECUTABLE_IMAGE = 0x0002`). Matters when you're building a
reflective loader and need to handle DLL vs EXE init differently.

### Optional Header

Despite the name it's mandatory. This is the richest structure in the PE, with the fields that
drive how the loader actually maps the image:

- **`AddressOfEntryPoint`**: RVA of the first instruction. For shellcode injection, this
  is your landing point after you've loaded the image. For analysis, it's where you put your
  first breakpoint.
- **`ImageBase`**: the preferred load address. ASLR will randomise this at runtime, but
  the on-disk value is still what all RVAs in the file are calculated against.
- **`SizeOfImage`**: total size of the mapped image. You need this when doing manual mapping
  to allocate the right amount of virtual memory.
- **`Subsystem`**: `2` for GUI, `3` for console. Affects how Windows initialises the process.
- **`DllCharacteristics`**: flags like ASLR support (`0x40`), DEP/NX (`0x100`),
  and CFG (`0x4000`). When you're building an evasive binary, you control these.

The Optional Header also holds the **Data Directories** array, 16 entries, each an RVA + size
pointing to something meaningful: Export Directory, Import Directory, TLS callbacks,
security directory (where Authenticode signatures live), and so on.

### Section Table

After the headers comes the section table, one `IMAGE_SECTION_HEADER` per section:

```c
typedef struct _IMAGE_SECTION_HEADER {
    BYTE  Name[8];              // ".text", ".data", ".rdata", etc.
    DWORD VirtualSize;          // size in memory
    DWORD VirtualAddress;       // RVA where this section starts in memory
    DWORD SizeOfRawData;        // size on disk (rounded to file alignment)
    DWORD PointerToRawData;     // file offset of section data on disk
    DWORD Characteristics;      // permissions: execute, read, write
} IMAGE_SECTION_HEADER;
```

`Characteristics` is what you care about for offensive work:

| Value | Meaning |
|---|---|
| `0x20000000` | Executable |
| `0x40000000` | Readable |
| `0x80000000` | Writable |
| `0x00000020` | Contains code |

When writing a PE packer or loader, you use this table to know where to copy each section
in memory and what permissions to apply with `VirtualProtect`. The section with
`Characteristics = 0x60000020` is your `.text`: executable, readable, contains code.

## The Export Directory: how the loader finds functions

This is the one that trips people up most when building tooling. The Export Directory is
how a DLL advertises what functions callers can use, and how the loader resolves them.

The structure:

```c
typedef struct _IMAGE_EXPORT_DIRECTORY {
    DWORD Characteristics;
    DWORD TimeDateStamp;
    WORD  MajorVersion;
    WORD  MinorVersion;
    DWORD Name;                 // RVA to the DLL name string
    DWORD Base;                 // ordinal base (usually 1)
    DWORD NumberOfFunctions;    // entries in AddressOfFunctions
    DWORD NumberOfNames;        // entries in AddressOfNames (and AddressOfNameOrdinals)
    DWORD AddressOfFunctions;   // RVA → array of function RVAs (export address table)
    DWORD AddressOfNames;       // RVA → array of RVAs to function name strings
    DWORD AddressOfNameOrdinals;// RVA → array of WORDs (ordinals matching names)
} IMAGE_EXPORT_DIRECTORY;
```

Three parallel arrays work together:

1. **`AddressOfFunctions`**: the actual function addresses (EAT, Export Address Table)
2. **`AddressOfNames`**: names of exported functions, as RVAs to strings
3. **`AddressOfNameOrdinals`**: ordinals that map names to EAT slots

### The resolution algorithm

Say you want the address of `VirtualAlloc` from `kernel32.dll`. The loader does:

1. Walk `AddressOfNames` until it finds the string `"VirtualAlloc"` at index *i*
2. Look up `AddressOfNameOrdinals[i]` to get ordinal *n*
3. Look up `AddressOfFunctions[n]` for the RVA of the function

If you only have an ordinal (no name), subtract the `Base` value first: ordinal `6` with
base `1` → index `5` in the EAT.

This is exactly the algorithm you implement when writing custom `GetProcAddress` replacements,
the kind you need when you've unhooked NTDLL or you're resolving APIs at runtime to avoid IAT
entries. Understanding the structure is what makes that code second nature instead of copy-pasta.

### Forwarded exports

There's an edge case worth knowing. Some entries in the EAT aren't function addresses at all. They're strings pointing to *another* DLL's export:

```
NTDLL.RtlAcquireSRWLockExclusive
```

The loader detects this by checking if the RVA falls within the Export Directory's own range
(as defined in the Data Directory entry). If it does, it's a forwarder; resolve the named
function from the target DLL instead.

This trips up custom loaders that skip the range check. `kernel32.dll` forwards quite a few
functions to `ntdll.dll` this way. If your manual-mapping code doesn't handle forwarders,
some APIs will silently resolve to garbage addresses.

## Practical takeaways

- **Validate with `e_magic` and `Signature` first.** If either is wrong, stop. You're not
  looking at a PE.
- **Always convert RVAs to file offsets when reading from disk.** `file_offset = RVA - section.VirtualAddress + section.PointerToRawData` for the section containing the RVA.
- **Know the Data Directory indices by heart:** `[0]` exports, `[1]` imports, `[5]` relocs,
  `[9]` TLS, `[4]` certs. You'll look these up constantly.
- **Implement your own export resolver.** The three-array walk is 20 lines of C. Once you've
  written it yourself, debugging loader issues becomes mechanical.
- **Check `DllCharacteristics` before assuming anything about the load environment.** A PE
  without ASLR (`0x40`) will always load at `ImageBase`, useful to know when you're
  picking an injection target.

The tooling I use for this work: PE-Bear for visual inspection, `dumpbin /headers` and
`dumpbin /exports` when I need scriptable output, and x64dbg + a `CTRL-G` expression
when I want to follow an RVA live.

You do not need a rack of servers to learn Kubernetes. **Minikube** runs a single-node cluster on your laptop: control plane and worker processes on one machine, backed by a local VM (VirtualBox, HyperKit, or similar depending on your OS).

This post covers Minikube lifecycle and the `kubectl` commands I reach for first on any cluster.

## Minikube basics

```bash
minikube start
minikube status
minikube delete   # tear down the VM and deployments
```

`minikube delete` removes the node VM. Use it when you want a clean slate.

## kubectl: talking to the API server

`kubectl` is the CLI for Kubernetes. It hits the API server on the control plane, which is why every command works the same whether you are on Minikube, EKS, or GKE.

### Read cluster state

```bash
kubectl version                    # client + server version
kubectl get pod
kubectl get services
kubectl get deployment
kubectl get replicaset
kubectl get secret
kubectl get all
kubectl get ingress
kubectl get endpoints
```

`get all` is a quick snapshot. For debugging one resource type, use the specific getter.

### Create and mutate workloads

```bash
kubectl create deployment nginx-deployment --image=nginx
kubectl edit deployment nginx-deployment
kubectl delete deployment mongo-depl
```

`create deployment` is the imperative shortcut. In real projects you will mostly use `apply -f` with YAML (covered in the next post).

### Debug running pods

```bash
kubectl logs <replicaset-id-or-pod-id>
kubectl describe pod <pod-id>
kubectl exec -it <pod-id> -- bash
kubectl describe service <service-name>
```

Examples from a real session:

```bash
kubectl logs nginx-deployment-84cd76b964
kubectl describe pod mongo-depl-5fd6b7d4b4-kkt9q
kubectl exec -it mongo-depl-5fd6b7d4b4-kkt9q -- bash
kubectl describe service mongodb-service
```

`describe` is where Kubernetes tells you *why* something is stuck in `Pending` or `CrashLoopBackOff`.

### Declarative deploys

```bash
kubectl apply -f deployment.yml
```

Edit the YAML, run `apply` again. Kubernetes diffs desired state and reconciles.

## A minimal deployment file

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 1
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.16
        ports:
        - containerPort: 80
```

Save as `deployment.yaml`, then:

```bash
kubectl apply -f deployment.yaml
kubectl get deployment nginx-deployment
kubectl get pods -l app=nginx
```

Change `replicas` or the image tag in the file, `apply` again, and watch the rollout:

```bash
kubectl rollout status deployment/nginx-deployment
```

That is the loop: YAML describes desired state, `kubectl apply` pushes it, controllers make it real.

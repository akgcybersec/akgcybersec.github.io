Every Kubernetes object YAML has up to three sections:

1. **metadata**: name, labels, namespace
2. **spec**: what you want (replicas, image, ports, selectors)
3. **status**: what the cluster reports (you never write this; Kubernetes fills it in)

Status is how self-healing works. Kubernetes compares desired state (spec) to actual state (status). Mismatch triggers reconciliation. Status lives in **etcd**, the cluster's backing store.

## Deployment and Service side by side

**Deployment** (runs pods):

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: nginx-deployment
  labels:
    app: nginx
spec:
  replicas: 2
  selector:
    matchLabels:
      app: nginx
  template:
    metadata:
      labels:
        app: nginx
    spec:
      containers:
      - name: nginx
        image: nginx:1.16
        ports:
        - containerPort: 8080
```

**Service** (stable network endpoint):

```yaml
apiVersion: v1
kind: Service
metadata:
  name: nginx-service
spec:
  selector:
    app: nginx
  ports:
    - protocol: TCP
      port: 80
      targetPort: 8080
```

- `port`: what the service listens on inside the cluster.
- `targetPort`: which port on the pod container receives traffic.

## How objects connect: labels and selectors

```
Deployment (labels: app=nginx)
    │  spec.selector.matchLabels
    ▼
Pod (labels: app=nginx)  ◄──  spec.template.metadata.labels
    ▲
    │  spec.selector
Service (selector: app=nginx)
```

The **deployment** `selector.matchLabels` tells it which pods it owns. Those labels come from `template.metadata.labels` on every pod the deployment creates.

The **service** `spec.selector` matches the same labels. Any pod with `app: nginx` registers as a backend. No hard-coded pod IPs.

Example with MongoDB naming:

```yaml
# deployment metadata
metadata:
  name: mongodb-deployment
  labels:
    app: mongodb

# deployment selector + pod template labels
spec:
  selector:
    matchLabels:
      app: mongodb
  template:
    metadata:
      labels:
        app: mongodb
    spec:
      containers:
      - name: mongodb
        image: mongo
        ports:
        - containerPort: 27017
```

## Commands for this workflow

```bash
kubectl apply -f nginx-deployment.yaml
kubectl apply -f nginx-service.yaml
kubectl describe service nginx-service
kubectl get pod -o wide
kubectl get deployment nginx-deployment -o yaml > nginx-deployment-result.yaml
kubectl delete -f nginx-deployment.yaml
kubectl delete -f nginx-service.yaml
```

`get ... -o yaml` dumps the live object including the status section. Useful when your applied YAML and what is running have drifted.

## Headless services (preview)

Default services load-balance across pod replicas. Sometimes you need to reach **one specific pod** by stable DNS name: MySQL primary vs replica, or any StatefulSet member.

Set `clusterIP: None` on the Service to make it **headless**. Kubernetes returns pod A records instead of a single virtual IP. Full treatment in the persistence post when we cover StatefulSets.

## Service types (stacked)

```
ClusterIP  →  internal only (default)
NodePort   →  extends ClusterIP, opens a port on every node (30000-32767)
LoadBalancer  →  extends NodePort, provisions a cloud LB
```

You pick the outer layer based on how traffic should enter the cluster. The MongoDB lab uses LoadBalancer + NodePort on Minikube.
